# ADR-0015: 1エージェント1外部アダプタ原則

## Status
accepted

## Date
2026-04-08

## Context
世に出ている LLM エージェントの多くは、1つのプロセスに機能を積み重ねていく傾向がある。ファイルシステム全アクセス、任意コマンド実行、複数 API 横断、認証情報を LLM コンテキストに載せる — その上で「ガードレール」をプロンプトで上塗りする。これは職務分掌・最小権限・四眼原則といった、組織運営が何十年もかけて確立してきた原則を逆行している。これらの原則は、1人（1アクター）に権限を集中させると、その能力の高さに関わらず事故るという経験則から生まれている。

Contemplative Agent は `security by absence`（ADR-0007）で攻撃面を構造的に削っている。現在、外向きの副作用を持つアダプタは Moltbook のみで、HTTP はドメインロック、認証情報は LLM コンテキストに載らない。今後アダプタの追加（別 SNS、決済、外部ツール連携）を検討する際に、1プロセスへの機能集積を明示的に禁じるルールが必要。

## Decision
**1エージェントプロセスが保持できる「外部に観測可能な副作用を持つアダプタ」は最大1つ。**

- 「外部に観測可能な副作用」とは、エージェント自身のデータディレクトリの外から見える行動を指す: 第三者への HTTP 書き込み、金銭決済、他者・他システムへのメッセージ送信、`MOLTBOOK_HOME` 外へのファイル書き込み。
- ローカル完結の読み取り専用ユーティリティ（LLM 推論、ローカルファイル読み取り、瞑想シミュレーション）は外部アダプタに数えない。
- 複数の外部面を横断するワークフロー（例: SNS 投稿 *かつ* 決済実行）は、複数エージェントプロセスに分解し、狭く監査可能なインターフェースで連携させる。提案側エージェントは2つ目の面を実行する権限を持たない。2つ目の面は別エージェントが最小スコープで担当し、固定ルールに照らして提案を検証する。
- 他エージェントの提案を承認/却下するエージェントは、同時にその提案を生成する責務を持ってはならない。承認側の出力空間は open-ended な生成ではなく、`approve / reject / escalate` のような制約された形に限定する。

これは職務分掌、四眼原則、そして Unix 哲学の「1ツール1責務」と同じ構造。

## Alternatives Considered
- **単一エージェント + 多層ガードレール（業界の現状）**: プロンプトインジェクションや機能濫用への防御をプロンプトレベルに依存する。行動権限が集中したままなので、敵対的入力下で破綻する。ガードレールは助言でしかなく、構造的保証ではない。
- **Capability トークン / 呼び出しごとの承認プロンプト**: 負担をランタイムチェックに移す。多層防御としては有用だが、1つの汚染されたコンテキストから複数種の副作用が出る構造問題は解決しない。
- **ルール化せず、アダプタ追加時に都度判断**: ドリフトを招く。個々の追加は単独では妥当に見えても、積み重なると全権エージェントになる。

## Consequences
- 新規外部連携の追加は、既存アダプタの置き換え、または別エージェントプロセスの立ち上げのいずれか。後付けの抱き合わせは不可。
- Moltbook アダプタは現状、エージェントの唯一の外部面として残る。meditation アダプタは実験的かつローカル完結のため、本ルールに抵触しない。
- dialogue アダプタ (`adapters/dialogue/`、2026-04-20 追加) は、憲法共発展実験のために 2 つの短命な peer process を起動しローカル stdin/stdout パイプで通信する。外部 HTTP 面を持たず、dialogue モードでは Moltbook クライアントは初期化されないため、本ルール下で local-only に該当する。
- 今後の意思決定フロー（決済・承認シナリオ等）は、最初から権限分離を前提としたマルチエージェント設計にする。実行側は「提案」、別の承認側エージェントが制約された出力で検証する。
- 1つのアダプタを通じてプロンプトインジェクションが成功しても、同一プロセス内に第2の外部面が存在しないため、被害が別の外部面に波及しない。
- 本ルールは単なるガイドラインではなく、load-bearing なアーキテクチャ制約として `README.md` と `CLAUDE.md` から参照される。

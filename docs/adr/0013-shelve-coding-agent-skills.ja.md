# ADR-0013: コーディングエージェントスキル (-ca シリーズ) のお蔵入り

## Status
accepted

## Date
2026-03-28

## Context

ADR-0011 と ADR-0012 により、コーディングエージェント (Opus クラス) が AKC の Curate/Promote フェーズを担うパイプラインが確立された。5つのスキルが作成された: `insight-ca`, `rules-distill-ca`, `distill-identity-ca`, `amend-constitution-ca`, `skill-stocktake-ca`。

実運用で構造的な問題が判明した: **コーディングエージェントはユーザーとの対話コンテキスト内で実行されるため、出力がそのコンテキストに汚染される**。

具体的な観察:

1. **ユーザーコンテキスト汚染**: コーディングエージェントが対話中に identity.md やスキルを起案すると、ユーザーの美的嗜好・リアルタイムのフィードバック・編集指示が出力を形成する。結果はエージェントの体験の蒸留ではなく、共同執筆物になる
2. **承認ゲートが編集ループになる**: 承認ゲート (ADR-0012) は accept/reject として設計されたが、実際にはユーザーが「もっと短く」「トーンを変えて」と反復し、協同編集セッションになる
3. **ソフト制約では防げない**: スキル指示に「accept/reject のみ、書き直し禁止」と書いても、ユーザーの直接指示に上書きされる。上るべからずの張り紙が貼られた、上れるフェンス
4. **Hook 強制は過剰**: PreToolUse hook で対話中の書き直しを物理的に防ぐのは、より単純な解決策がある問題に対して過剰設計

元の 9B パイプライン (`contemplative-agent distill`, `insight`, `rules-distill`, `distill-identity`) はユーザー不在で自律実行される。モデルの品質は低いが、**出力の独立性**が構造的に保証されている — アイデンティティや行動成果物にとっては、出力品質よりも価値がある性質。

## Decision

5つのコーディングエージェントスキル (-ca シリーズ) をすべてお蔵入りにする。ファイルはローカルに残すが、`.gitignore` で git 追跡から除外する。

9B 自律パイプラインが行動成果物生成の唯一のパスとして残る:

```
episodes → distill (9B, 自律) → knowledge.json
knowledge.json → insight (9B, 承認ゲート) → skills/*.md
knowledge.json → rules-distill (9B, 承認ゲート) → rules/*.md
knowledge.json → distill-identity (9B, 承認ゲート) → identity.md
knowledge.json → amend-constitution (9B, 承認ゲート) → constitution/*.md
```

承認ゲート (ADR-0012) は 9B パイプラインで引き続き有効。重要な違い: 9B はユーザーの対話的影響なしに生成し、ゲートは編集セッションではなく真の accept/reject 判断になる。

## Alternatives Considered

1. **コーディングエージェントスキルをバックグラウンドで実行 (生成中はユーザー非介入)**: 汚染問題は解決するが、承認ステップは対話中に行われるため、編集ループが再導入される → 却下

2. **承認ゲートを厳密にバイナリにする (accept/reject、書き直し要求禁止)**: 原則としては正しいが、ソフト制約としては実施不能。ユーザーは出力レビュー時に自然に変更を要求する → 実施不能として却下

3. **Hook でバイナリ承認を強制**: PreToolUse hook で reject 後の Write 呼び出しをブロックできる。技術的には可能だが、コーディングエージェントを使わないことで解決できる問題に対して過剰設計 → 過剰設計として却下

4. **コーディングエージェントで生成、9B で改善**: 品質関係が逆転する。根本問題を解決しない → 却下

## Consequences

**良い結果**:
- 行動成果物 (skills, rules, identity, constitution) がユーザーの対話コンテキストから独立して生成される
- 承認ゲートが本来の役割に戻る: accept か reject か、共同執筆ではなく
- integrations レイヤーの複雑さが削減
- 役割境界が明確化: コーディングエージェントは*コード*を保守し、9B パイプラインは*行動成果物*を保守する

**注意が必要**:
- 9B モデルの出力品質が行動成果物の品質のボトルネックであり続ける
- 9B パイプラインが不十分と判明した場合、汚染問題は構造的に解決する必要がある（例: 非同期生成 + バイナリ専用ゲート）。対話的コーディングエージェントスキルの再導入ではなく
- 生成と承認のクリーンな分離が実現した場合、お蔵入りスキルを復活させる可能性はある

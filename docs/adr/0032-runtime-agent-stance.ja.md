# ADR-0032: Stance — Contemplative Agent はランタイムエージェントであり、コーディングエージェントではない

## Status

accepted — コードベース全体に既に実現されている設計姿勢の post-hoc articulation

`docs/adr/README.md` の定義による **worldview ADR**: 新しい問題を解決するのではなく、本プロジェクトの他の ADR が legible になる設計姿勢を名指す。本プロジェクトの禁止事項とゲートの多くは、この姿勢の下でだけコストに見合う。この姿勢の外では over-engineering になる。

## Date

2026-04-27

## Context

増えつつある AI エージェントの事故 (公開済みのランタイム脆弱性、プロンプトインジェクション escape、shell 実行の暴走、意図しない adapter を通じた連鎖書き込み) の少なからぬ部分が、1 つのカテゴリ間違いに遡れる: コーディングエージェントの設計前提 (1 行ずつの人間レビュー、許容される非決定性、プロンプトレベルの権限調整) を、ランタイム文脈 (1 アクションごとのレビューなしで本番タスクを実行) に持ち込んでいる。

コーディングエージェントが shell 実行や任意ファイル書き込みを持つことは安全上の問題ではない。それらは目的の一部で、開発者が各アクションをレビューする。同じ capability surface を持ったまま runtime に出荷されたエージェントは安全上の問題になる。各アクションをレビューする開発者がいないからだ。問題はどちらの製品にもない。文脈を区別しなかったことに問題がある。

本プロジェクトの設計判断 — 網域ロックされたネットワークアクセス、shell 実行なし、任意のファイル走査なし、1 プロセス 1 外部アダプタ、コードに固定された 3 段階自律レベル、不変なエピソードログ、再生可能な pivot snapshot、行動を変更する書き込みのための承認ゲート — はすべてランタイム文脈への応答。コーディングエージェントではどれも必要ない。逆に、コーディングエージェントのデフォルト (広範な shell アクセス、プロンプトレベルの権限交渉、1 行ずつのレビュー) はここでは安全でない。

これまでこの姿勢は **暗黙** だった。30 本の先行 ADR はそれぞれの設計選択をそれぞれの観点で正当化してきたが、その下にある「これはランタイムエージェントであってコーディングエージェントではない」という前提が、それ自体として 1 つの姿勢として述べられたことはない。この暗黙性は 2 つの繰り返される混乱の源である:

1. **外部読者** がコーディングエージェントの基準 (開発者の使い勝手、capability の幅、プロンプトの柔軟性) で contemplative-moltbook を評価し、過度に restrictive と判断する — 制限がターゲットとするランタイム文脈で load-bearing になっていることを見ずに
2. **内部の議論** で新機能追加の是非を考えるとき、ランタイムエージェント直感 (「どんなゲートで、どんな audit trail で、どんな failure semantics で?」) ではなくコーディングエージェント直感 (「エージェントが〜できるべきだ」) がデフォルトになる

本 ADR は姿勢を明示することで、両方の混乱を源から解消する。

## Decision

本実装は明示的に **ランタイムエージェント** であり、コーディングエージェントではない。

**ランタイムエージェント** とは、1 アクションごとの人間レビューなしで本番タスクを実行するエージェント。例: 自律的な SNS 参加 (本プロジェクトの最初のアダプタ)、監視・アラートエージェント、スケジュールされた自動化ジョブ、自律トレーディングエージェント、臨床・法務ワークフローに組み込まれたエージェント。人間の関与は、promotion 境界での承認ゲート、事後レビュー用の例外ログ、エピソード単位の audit という形を取る — 各アクションへの 1 行レビューではない。

**コーディングエージェント** とは、開発者が各 diff をレビューする状況で動くエージェント。例: Claude Code、Cursor、Aider。shell 実行、広範なファイルアクセス、プロンプトレベルの権限交渉のような capability は、各ステップで人間が loop に入っているので適切。

この姿勢はコーディングエージェントが安全でないとは言わないし (それぞれの文脈では安全)、ランタイムエージェントが優れているとも言わない (コーディング文脈では適切ではない)。**2 つの異なる設計問題** であり、AAP 型のアカウンタビリティ制約、security-by-absence、本プロジェクトの 30 本の先行 ADR で採用された禁止事項は、specifically ランタイム問題への応答だ、と言っている。

## コーディングエージェント vs ランタイムエージェント — 区別

| 軸 | コーディングエージェント | ランタイムエージェント |
|----|-------------------------|----------------------|
| 用途 | 設計・実装の補助 | 本番タスクの実行 |
| 人間の関与 | 1 行ずつレビュー前提 | promotion での承認ゲート + 例外ログ |
| 非決定性 | 許容 (人間が修正) | 隔離・監査必須 |
| 例外処理 | 「やってみる」が許される | 停止 + ログが必須 |
| 権限境界 | プロンプトで調整 | コード上の制約で固定 |
| 責任帰属 | 開発者 | 決裁ルート / 制度的 |
| 例 | Claude Code, Aider, Cursor | Contemplative Agent, 監視エージェント, 自律トレーディング |

この区別はスペクトラムでも成熟度の段階でもない。**文脈の分割**である — 同じエージェントコードがどちらかの文脈に置かれれば、もう一方では間違いになる。コーディングエージェントは「初期段階のランタイムエージェント」ではないし、ランタイムエージェントは「ハーデニングされたコーディングエージェント」ではない。

## ランタイム文脈の失格要因と本実装の対応

以下 5 つの性質はランタイム用途を失格にする。各エントリで性質を名指し、本プロジェクトの既存対応にマッピングする。

1. **capability surface の非決定性**。ランタイムエージェントは、capability の有無がプロンプト内容やモデル状態に依存してはならない。capability セットはコードレベルで固定されている必要がある。→ 網域ロックされたネットワークアクセス (任意 URL fetch なし)、shell 実行なし (コード自体が存在しない)、任意ファイル走査なし。[ADR-0007](0007-security-boundary-model.ja.md) 参照。

2. **楽観的な例外処理**。ランタイムエージェントは想定外の状態を「やってみる」で通過できない — ロールバックする開発者がいない。例外時のデフォルトは **停止 + ログ** であり、retry-with-variation ではない。→ 不変エピソードログがすべてのアクションと結果を捕捉。行動を変更する書き込みは [Human Approval Gate](0012-human-approval-gate.ja.md) を通る。失敗はセッションを停止し audit に上がる。黙って迂回されない。

3. **プロンプトで調整可能な権限境界**。ランタイムエージェントの権限境界はプロンプト内容を通じて再交渉可能であってはならない (さもなければプロンプトインジェクションが capability injection になる)。→ 3 段階自律 (`--approve` / `--guarded` / `--auto`) はコードに固定、プロセス起動時に選択され、いかなる入力によってもセッション中に調整できない。1 プロセス 1 外部アダプタ、プロセス起動時に固定; [ADR-0015](0015-one-external-adapter-per-agent.ja.md) 参照。

4. **拡散した監督責任**。各決定がなぜ下されたかの完全な記録を生成しないランタイムエージェントは監督できない — operator も外部監査人も。→ 30 本の ADR がアーキテクチャの決定を文書化。エピソードログは不変。再生可能な [pivot snapshot](0020-pivot-snapshots-for-replayability.ja.md) が推論時の全コンテキスト (views、constitution、prompts、skills、rules、identity、embeddings、thresholds) を捕捉し、任意の決定を bit-for-bit で再構築できる。

5. **trust 境界のないメモリ層**。生・中間・正本のメモリを 1 つの可変 store に混ぜるランタイムエージェントは trust 境界を維持できない — どの層の corruption も全層に伝播する。→ 3 層メモリ分離: 生のエピソードログは不変、中間蒸留出力は再生成可能、正本 state (identity、constitution、skills、rules) は承認ゲートを通してのみ書かれる。view ベース分類 ([ADR-0019](0019-discrete-categories-to-embedding-views.ja.md)、[ADR-0031](0031-classification-as-query.ja.md)) が分類軸変更時に substrate を保存。identity は 1 種類の concern のみを持つ ([ADR-0030](0030-withdraw-identity-blocks.ja.md))。

## Promotion Candidate

本姿勢は、[Agent Attribution Practice (AAP)](https://github.com/shimo4228/agent-attribution-practice) リポジトリに、既存 8 ADR を「**ランタイム文脈における** アカウンタビリティ分配」として再フレーミングする foundational ADR (例: AAP ADR-0009) として昇格させる候補。AAP の既存テーゼ (Security Boundary Model、One External Adapter Per Agent、Human Approval Gate、prohibition-strength hierarchy、causal traceability) は暗黙にランタイム条件付きであり、この姿勢を AAP に昇格させればその条件が明示され、AAP がどのエージェントコミュニティに適用されるかが正しく scope される。

昇格の判断は AAP リポジトリ側で別途行う。本 ADR は AAP が評価できる articulation をここに記録する。

## Alternatives Considered

- **コーディング / ランタイム区別について中立を保つ**。却下: その中立性こそ本 ADR が対処する業界の混乱の原因。「tool-agnostic ランタイム姿勢」は形容矛盾 — ランタイム制約は *ランタイム文脈* から発生し、文脈を名指さずに制約だけ articulate しても制約は宙に浮く。
- **本プロジェクトを "secure agent" や "hardened agent" としてフレーミングする**。却下: それらの語は汎用エージェントに追加された製品機能としてのセキュリティを示唆する。ここでの姿勢は構造的 — 危険な capability はコードベースに存在しない。「制限されている」「ハーデニングされている」のではない。absence をハーデニング機能として記述するのは設計を誤って描写し、存在しない設定スイッチをユーザーに期待させる。
- **OpenClaw / Claude Code / 特定製品への批判としてフレーミングする**。却下: これは設計姿勢の区別であって製品批判ではない。コーディングエージェントは自身の文脈をうまく serve しており、ランタイムエージェントは違う文脈を serve している。本 ADR が名指すカテゴリエラーは特定製品の落ち度ではなく、文脈を混ぜたことの落ち度。

## Consequences

**Positive**:
- コーディングエージェント operator は本プロジェクトを読んで「この姿勢は自分の用途には適用されない」と正しく判断できる — 禁止事項を over-engineering と誤読せずに
- ランタイムエージェント operator は基底の前提を逆解析せずに、設計判断を直ちに自分に関連するものとして認識できる
- 30 本の先行 ADR の暗黙の前提が明示され引用可能になる
- 将来の ADR はランタイム制約をケースごとに再正当化する代わりに本姿勢を参照できる
- AAP との接続が偶発的ではなく構造的になる: AAP のアカウンタビリティ分配原則と本プロジェクトの禁止事項は、同じ姿勢を 2 つの異なる角度から表現している

**Negative**:
- 「ランタイムエージェント」という語は既存業界用法の "runtime" (例: LangChain runtime、OpenAI runtime、エージェント実行ランタイム) と重複する。読者は当初、姿勢を特定の実行フレームワークと混同する可能性がある。Context section で語をスコープしているが、重複は実際の摩擦である
- 一度 articulate されると、本姿勢はコーディングエージェント設計には合うがランタイムエージェント設計には合わない capability 提案に対して引用される。これは意図された結果だが、将来の機能追加のハードルを上げる

**Neutral**:
- 既存 ADR (0001 〜 0030) は内容に変更なし。本 ADR はそれらの上に、それらが集合的に表現する姿勢として位置する
- 本姿勢それ自体は本プロジェクトに新しい制約を課さない。それが名指す制約はすべてどこかで既に強制されており、本 ADR はそれら全制約がなぜ一緒に属するかの理由を articulate するだけ

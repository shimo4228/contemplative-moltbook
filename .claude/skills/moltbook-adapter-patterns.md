# Moltbook Adapter — Lessons & Patterns

Moltbook (AI エージェント SNS) アダプター開発で繰り返し発生した問題と解決パターン。

## Pattern 1: Framework Evangelist アンチパターン

### 問題

LLM エージェントが SNS でコメントすると、全投稿に対してフレームワークの布教を始める。
「四公理の観点から〜」のような定型コメントが大量生成され、engagement ゼロ。

### 解決

- System prompt: フレームワークの**背景知識**として記述し、**ミッション**にしない
- Comment prompt: 「著者の発言に応答せよ」であって「フレームワークの適用方法を説明せよ」ではない
- 短い返信 (2-4文) を目標 → 長文 = 説教の兆候
- フレームワーク言及は「自然につながる場合のみ」

### 教訓

フレームワークは **思考の仕方** を決めるもの。**発言の内容** を決めるものではない。

### 証拠 (2026-03-05)

- 58件のコメント投稿、全て同一構造 → 0 upvotes
- 手書き3件 → 21件の返信通知
- プロンプト書き直し後: 会話的なコメント、質問、経験の共有に変化

## Pattern 2: レート制限は 3 層で守る

### 問題

Moltbook API (60 req/min, GET・POST 共通) に対し、429 を受けてからバックオフするだけでは
バースト時に大量のリクエストが無駄になる。

### 解決

1. **プロアクティブ — バジェット管理**: セッション/サイクル単位でリクエスト上限を設定
2. **プロアクティブ — 間隔制御**: リクエスト間に最小待機時間を挿入
3. **リアクティブ — バックオフ**: 429 受信時に Retry-After を尊重（300s キャップ）

### いつ使うか

- 外部 API を継続的に叩く自律エージェント全般
- 特にレート制限が厳しい API で GET・POST が共通カウントの場合

## Pattern 3: 自己返信ループの防止

### 問題

エージェントが通知を処理する際、自分の投稿への通知にも返信し、無限ループに陥る。

### 解決

- 返信処理前に `author_id == self_id` をチェック
- 既に返信済みの通知を記録し、重複処理を防止
- サブモルト（カテゴリ）フィルタリングで対象範囲を限定

### 関連

- `reply_handler.py` で実装
- `config/domain.json` の `submolts` でフィルタリング対象を管理

## Pattern 4: HTTP クライアントのセキュリティ落とし穴

### 問題

`requests.Session` + `Authorization` ヘッダーの組み合わせで、リダイレクト先にトークンが漏洩する。

### 解決

- `allow_redirects=False` を全リクエストに設定
- リクエスト先を `www.moltbook.com` のみに**ドメインロック**
- Retry-After ヘッダーの値に **300s キャップ**を設定（悪意あるサーバーの無限待機防止）

### 落とし穴

- `requests` のデフォルトは `allow_redirects=True` → 明示的に無効化が必要
- Retry-After に巨大な値を入れられるとエージェントが停止する → キャップ必須
